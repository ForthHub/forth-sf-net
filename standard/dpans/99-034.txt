------ the question ----------------------------------------------------

What must happen if file operations are used with the value returned
by SOURCE-ID ?
How INCLUDE-FILE and INCLUDED interfere with other file access words?

In other words:
>11.6.1.1718 INCLUDED 				FILE
>	( i*x c-addr u -- j*x )
> ...
>Repeat until end of file: read a line from the file, fill the input
>buffer from the contents of that line, set >IN to zero, and interpret.
> ...

Does the word "read" imply the functionality of READ-LINE ?
At the conclusion of the operation ("read"), what the value returned
by FILE-POSITION shall be?
Are implementors allowed to let their systems read the whole file
into a buffer?

------------------------------------------------------------------------

The ambiguity springs from the fact that INCLUDE-FILE may read lines
one by one or load the whole file into a buffer and take the lines
from there.

I see several (currently, 5) possible variants of the answer, the best
one IMHO is the 4th. Note that the variants 1-4 allow the use of buffers
for the files being INCLUDEd, while the 5th does not.

---- VARIANT 1
----------------------------------------------------------------
> 11.6.1.2218 SOURCE-ID 			source-i-d 	FILE
>	( -- 0 | -1 | fileid )
> ....
Ambiguous conditions:
- fileid is used with other file access operations
- a program changes the file specified by fileid via a different file
  identifier

Rationale. The system implementors are free to read the source file
line-by-line or to load it into a buffer (the whole file or a part of
it). The position at which the file pointer will be, as well as the
effects of changing it, are unpredictable in the general case.

---- VARIANT 2
----------------------------------------------------------------
> 11.6.1.2218 SOURCE-ID 			source-i-d 	FILE
>	( -- 0 | -1 | fileid )
> ....
Ambiguous conditions:
- a program performs a file access operation on the file specified by
  fileid using the current position of the file
- a program changes the current position of the file specified by
  fileid and does not restore it before the system attempts to refill
  the input buffer
- a program changes the file specified by fileid

Rationale. The system implementors are free to read the source file
line-by-line or to load it into a buffer (the whole file or a part of
it). The position at which the file pointer will be, as well as the
effects of changing it, are unpredictable in the general case.

Note: the input buffer may be refilled at the end of line or due to
execution of RESTORE-INPUT or other word temporary changing the input
source.

---- VARIANT 3
----------------------------------------------------------------
> 11.6.1.2218 SOURCE-ID 			source-i-d 	FILE
>	( -- 0 | -1 | fileid )
> ....
Ambiguous conditions:
- a program performs a file access operation on the file specified by
  fileid using the current position of the file
- a program changes the current position of the file specified by
  fileid and does not restore it before the system attempts to refill
  the input buffer.
- a program changes the already existing portion of the file
  specified by fileid
- a program changes the file specified by fileid via a different file
  identifier

Rationale. The system implementors are free to read the source file
line-by-line or to load it into a buffer (the whole file or a part of
it). The position at which the file pointer will be, as well as the
effects of changing it, are unpredictable in the general case.
On the other hand, if new text is added to the end of file, the
system stil can determine it and interpret the text.

Note: the input buffer may be refilled at the end of line or due to
execution of RESTORE-INPUT or other word temporary changing the input
source.

---- VARIANT 4
----------------------------------------------------------------
> 11.6.1.2218 SOURCE-ID 			source-i-d 	FILE
>	( -- 0 | -1 | fileid )
> ....
The current position for the file identified by fileid will correspond
to the beginning of the next source line (the line immediately following
the system-defined enf-of-line indicator of the line being currently
interpreted with the text interpreter).

Ambiguous condition:
- a program changes the line being currently interpreted in the file
  specified by fileid.
- a program changes the file specified by fileid via a different file
  identifier

Rationale. The system implementors are free to read the source file
line-by-line or to load it into a buffer (the whole file or a part of
it). In the latter case, the system must check if the fileid passed
to the file access operations corresponds to one of the files being
currently loaded (INCLUDEd), and behave as if the file was read
line-by-line. If the program changes the file specified by fileid,
the contents of the buffer must be adequately changed. Implementing
a buffer for file loading is equivalent to implementing a cache.
Reading/writing to the cached file bypassing this cache (e.g. by
means of operating system of firmware) also leads to
unpredictable results, but the only "standard" way of bypassing the
cache is using two or more different file ideentifiers for the same
file. The statement that changing the line being currently interpreted
is ambiguous means that the systems (especially those that read files
line-by-line) are allowed not to reload the current line when the file
changes, but this may happen as a result of execution of SAVE-INPUT
and RESTORE-INPUT.

---- VARIANT 5
----------------------------------------------------------------
All what is written above is wrong (below ":" marks citations
from  'variant 4'). We see in 11.6.1.1718 INCLUDED :

>11.6.1.1718 INCLUDED 				FILE
>	( i*x c-addr u -- j*x )
> ...
>Repeat until end of file: read a line from the file, fill the input
>buffer from the contents of that line, set >IN to zero, and interpret.
> ...

The words "read a line" above mean "read as with READ-LINE", not
"get somehow the next line into the input buffer, for example,
pre-read the whole file into a buffer and then take a line from
that buffer".
This means that a standard system is not allowed to read the whole
file into a buffer and then take lines from there. As a result,

:The current position for the file identified by fileid will correspond
:to the beginning of the next line (the line immediately following
:the system-defined enf-of-line indicator of the line being currently
:interpreted with the text interpreter).

and two comditions
:Ambiguous condition:
:- a program changes the line being currently interpreted in the file
:  specified by fileid.
:- a program changes the file specified by fileid via a different file
:  identifier
should be, indeed, ambiguous. The first one is ambiguous because
a standard system may or may not reload the current line when the file
is changed and the words SAVE-INPUT and RESTORE-INPUT execute.
The second one should be ambiguous because the result would depend
on the operating system and loaded drivers.

Note that variants 4 and 5 are equivalent.

===============  2.  TC Reply to Q0005.  ===========================

This RFI raises several related questions.  Before addressing them,
the following general remarks are relevant.

The TC consciously revealed the fileid's involved in file based
interpretation because this was common practice at the time and
existing implementations had made the fileid or equivalent available
for judicious application use.  Some uses are consistent with the
intent of the Standard.  Others are not, and as always the standard
has made no effort to enumerate every possible mode of abuse.

For example, it is obvious that while the fileid given INCLUDE-FILE
is certainly known to the application, it would be imprudent for the
application to use CLOSE-FILE on that fileid during its use by the
interpreter, and the fact that the Standard does not explicitly
forbid this action does not in any way diminish its absurdity.

In this light the individual points are addressed as follows:

> What must happen if file operations are used with the value returned
> by SOURCE-ID ?

Passive file operations such as FILE-POSITION and FILE-SIZE are
appropriate.

Reading and repositioning operations are appropriate and were
specifically envisioned by those who drafted the file word sets,
although these activities are of course environmentally dependent
upon having a file as the input source and in many cases (such as
sequential files like tapes, or temporally disjoint files such as
terminal devices, named pipes, or network connections) the very
ability to reposition is environmentally dependent on the *type*
of file.  It is recognized that the ability to perform these oper-
ations constrains implementations, but such practices were common
and hence no effort was made to prohibit them.

The "synchronization" of the text interpreter with other explicit
file operations opens the door to useful techniques such as files
that begin with Forth code whose purpose is to operate on data that
follows the code in the file.

Any type of modification operations on the file itself,  such as
writing, resizing, or deletion, and on the fileid, such as closing,
is either prima facie absurd (such as closing) or depends on "carnal
knowledge" of the implementation and is hence environmentally depen-
dent on things the Standard doesn't require that implementations
document.  Furthermore the ability to write into a "file" is in
itself environmentally dependent on the characteristics of the file
and the way in which it was opened.  Such operations may be useful
but are not portable and are beyond the scope of the Standard.

As a decision making example, it is without question a *useful*
technique for one part of a program to inject source code into a
named pipe while another part is interpreting from the other end
of that pipe.  However a program that depends on operating system
support for named pipes is certainly not standard!

> How INCLUDE-FILE and INCLUDED interfere with other file access words?
>
> In other words:
> >11.6.1.1718 INCLUDED 				FILE
> >	( i*x c-addr u -- j*x )
> > ...
> >Repeat until end of file: read a line from the file, fill the input
> >buffer from the contents of that line, set >IN to zero, and interpret.
> > ...
>
> Does the word "read" imply the functionality of READ-LINE ?

Yes, that is exactly what it means.

(11.6.1.2090, the definition of READ-LINE, is the only definition
the standard offers for "reading a line from a file" and is by
definition what the phrase cited above refers to.  "line" is
defined in 2.1 in a way that is consistent with 11.6.1.2090.

> At the conclusion of the operation ("read"), what the value returned
> by FILE-POSITION shall be?

Refer to 11.6.1.2090 READ-LINE which defines this value.

> Are implementors allowed to let their systems read the whole file
> into a buffer?

Yes.  Many systems cache files in memory.

> The ambiguity springs from the fact that INCLUDE-FILE may read lines
> one by one or load the whole file into a buffer and take the lines
> from there.

There is no ambiguity.  An implementor may cache a file if desired
but must implement file caching in such a way that it is still
accessed through the normal file operators.  This is common practice
throughout the computer industry.  Caching or any other optimization
that is done in a transparent way is entirely consistent with the
Standard.

An ambiguity only arises if an implementor forgets that transparency
is required of him, which would be an error since transparency is
a paramount concern when conforming to standards or seeking portability.

MLG eMail: Files, SOURCE-ID's and the Interpreter
=================================================
Steve Coul, sc@mpeltd.demon.co.uk


Introduction
============
From what I can gather from MLG's eMail the problem is the relationship
between the Interpreter, the File sub-system and the implementation of
INCLUDE-FILE. Other comments in this document address the issue of
writing interpreter extensions in a portable manner, which at present
appears to be impossible without carnal knowledge of the underlying
system.

This highlights a deficiency of the existing specification.
INCLUDE-FILE hints strongly in it's description that it can (or at
least should) be implemented using other freely available factors
within ANS.

        1) The act of reading the next source line and resetting >IN
           is described by REFILL.

        2) The 'intepret line until parse area is empty' bit is
           remarkably similar to EVALUATE (except for the fact that
           EVALUATE resets SOURCE-ID)

Unfortunatly it also requires carnal knowledge. This fact makes it
harder to predict or standardise its' behaviour since it is not a
*closed* word, i.e do something with no possibility for side effects.
The act of INCLUDE-FILE is to modify the behaviour of the interpreter
and compiler and accept further input in the modified manner. Since
it is what I would call an *open* action it should be very tightly
defined within the standard abilities of the system, and not modify
the functionality of the interpreter. I don't like the idea of having
a system which behaves differently depending on the input source.

An implementation based on these ideas has been developed and tested.


Comments on MLGs "Variants"
===========================

1)      Making the use of the FILE-ID as returned by SOURCE-ID
        ambiguous is a break in the standard. If you cannot use
        the return result for file-access then it is not a FILE-ID.

2)      If you can't trust the current FILE-POSITION, cannot modify
        the file, and cannot change the position then you can't actually
        do anything to the file, so again SOURCE-ID is not really
        returning a FILE-ID, unless ANS is extended to allow for
        *locked* files, along with the management overhead in any
        implementation that wished to be standard.

3)      Ditto

4)      As MLG states at the end - this variant breaks SAVE/RESTORE
        input. I don't believe the TC should stand a wordset which
        breaks another wordset.

5)      MLG states this is equivalant to Variant 4 - It is not. Variant
        5 is the nearest to safe I can see. Caching *is* possible since
        there is no reason why READ-LINE cannot act on a memory-image
        despite MLG's comments; indeed under the hood almost any
        operating system does it anyway, the FILE-ID does *not* have
        to be an operating system supplied file-handle, you may not
        even have an operating system. FILE-ID is another ANS opaque
        type, and may be a pointer to a structure which maintains the
        cache information.

        The trick is to cache the FILE and not the INCLUDE. The only
        downside of this is the management overhead for R/W
        INCLUDE-FILES. I would actually like to see INCLUDE-FILE
        specify the requirement for files to be opened as R/O.


My Solution To All Problems ;-)
===============================

1)      Assume the existence of INTERPRET-SOURCE or similar, which
        would be worth standardising in order to permit portable
        extensions that have defined behaviour.

        INTERPRET-SOURCE        \ i*x -- j*x
        Interpret the current parse area (described by SOURCE in ANS)
        until empty. In effect, this is a factor of EVALUATE which
        does not of itself affect the value of SOURCE-ID.

2)      It should be stated that REFILL when operating on a FILE will
        act in the same manner as READ-LINE (regardless of its' actual
        implementation). Although it would be desirable to move REFILL
        from FILE-EXT to FILE, all source file words should be in the
        same word set so that file handling can be implemented on
        embedded systems without requiring source file handling.

3)      Add a new word to CORE or CORE-EXT

        IS-SOURCE-ID    \ fileid --
        Set the value returned by SOURCE-ID.

4)      INCLUDE-FILE and INCLUDED should be moved to FILE-EXT and
        documented as behaving according to the use of REFILL and
        INTERPRET-SOURCE.

        During interpretation of a source-line the SOURCE-ID will
        then be valid with the file pointer at the beginning of the
        *next* line.

        Document an INCLUDE-FILE in the Appendix of the standard as
        using something like the following code (see below for
        definitions of N>R and NR>):

        : include-file          \ i*x fileid -- j*x

          Save-Input N>R

          IS-SOURCE-ID          \ set SOURCE-ID

          begin
            refill
          while
            ['] interpret catch
            dup if
              source-id close-file drop
              NR> Restore-Input
            then
            throw
          repeat

          source-id close-file drop
          NR> Restore-Input
        ;

This will allow for cached systems and allow the predictable use of
file-access words during an INCLUDE.

With the increase in the number of words which return an
implementation defined number of items with the number of items
on the top of the stack, e.g SAVE-SECTIONS and RESTORE-SECTIONS
in the proposed cross compiler word set, it would make sense to
define N>R and NR>.

        N>R             \ x1..xn u -- l ; R: -- n x1..xn | n xn..x1
        Save u items and u on the return stack. The order in which
        x1..xn are saved on the return stack is implementation
        dependent, except that NR> must restore x1..xn u.

        NR>             \ R: x1..xn u | xn..x1 u ; -- x1..xn u
        Restores the data previously saved by N>R

Stephen,

As the SOB who drafted the response in question last Fall, I'm
appreciative of your observations and regret that the only feedback
I received on the matter after posting it was from Mitch, whose
advice I took.

I agree that it would be tidy to specify R/O opening of the file
but certainly not in the context of an RFI response.  Assuming
that this RFI is adopted it will call for an implementing proposal
at which time substantive change is feasible.  I believe that
the genesis of this function, as with the rest of the FILE word
set, was the synthesis worked out by those who actually supported
file system operations way back in the earliest meetings of the
original TC.  It was apparently common practice to have such a
function that laid responsibility for obtaining the file handle
upon the application, hence the distinction between INCLUDED and
INCLUDE-FILE.  It is my impression that we felt there was still
some variation in bufffering methods used when interpreting from
a file and perhaps a few years later this could be more tightly
specified without damaging applications that depend on any
particular synchronization.  If certain about that I'd love to
see it tighter myself.

Of course, if one is tighter about it, then there is no reason
to specify R/O.  The problematic aspect of tightness seemed to
center around handling of cases when the interpreter needed to
REFILL but I believe most of the discussions in which it seemed
that way preceded the definition of REFILL.  If by now everyone
is doing things in consonance with such facilities, the ambig-
uity might well no longer exist.  Initially, though, there were
really some wide differences in practices folks were describing.

I personally do not think that usage restrictions have no place
in the standard.  The world of programming abounds with usage
restrictions.  If my part of the program is using the variable
at an  addr  for something then yours is not entitled to just
step on my data whenever you like.  This does not make the
address of the variable any less an  addr  in my opinion.  It
just means that there are rules which we must both respect
governing our mutual use of such an address.  Similarly, in the
case of  INCLUDE-FILE  I as an application have a fileid for
which I am responsible.  When I present this  fileid  to the
INCLUDE-FILE function, I am in essence making a contract with
that word.  I am promising that the fileid is valid and that
the file thus described is full of good text to interpret.
I am also promising not to rip the file out from under its
feet until INCLUDE-FILE says it's done with it by returning.

If I break any aspect of this contract, various things can
happen.  In some, but certainly not all, cases I might catch
an exception.  However in response to the particular matter
you raise:

> On The TC's response
> ====================
>
> What is so absurd about closing the source-file during an INCLUDE?
> Since the INCLUDE almost has to use REFILL (otherwise why extend it
> to work with files?) then the act of closing the source file will
> cause REFILL to fail and should thus cleanly stop compilation from
> that source. Sounds neater than using CATCH/THROW when you don't
> really mean it to be an error, doesn't it?

11.3.1.1  says that a filed is invalid after it is closed.

11.4.1.2  says that invalid fileid in an INCLUDE-FILE is an
          ambiguous condition.

3.4.4  lists allowable actions on an ambiguous condition.

While throwing an exception is among the allowable options,
so is taking "implementation dependent" actions.  Among others,
this class was specifically designated to cover cases such as
an underlying operating system whose action in some exceptional
cases is to crash, or to terminate the program with a memory
dump.  It covers cases such as a portable Forth written in a
high level language to be run on unknown operating systems and/or
run time libraries that might vary in their handling of some
exceptions.

In the context of an RFI, which is simply clarifying what the
Standard currently says, I would have to say that intentionally
taking an action which is guaranteed to produce an ambiguous
condition which may have nondeterministic results is certainly
absurd.  I disagree that signaling an exception to an outer layer
of code by forcing an ambiguous condition is cleaner than is
throwing an exception.  We added CATCH/THROW specifically to
be *the* clean way to abort an operation without uncontrolled
side effects.

Anyway, that is my justification for having used that word
to describe the subset of things one could do with the file
handle that could produce dramatic nondeterministic results,
as opposed to writing into the file which might produce effects
not quite as expected but less likely to involve fireworks,
as the standard in question is written.


I'm writing privately because your comments are well taken
and we will certainly be discussing this later if this RFI
is adopted and needs to be implemented.  In particular I'd
like to if possible find out how widely common practice has
converged upon using the FILE operators directly within the
interpreter since if that's happened we could do a lot with
simple tightening of existing specification without creating
a lot of extra mechanism.

# This RFI raises several related questions.  Before addressing them,
# the following general remarks are relevant.
#
# The TC consciously revealed the fileid's involved in file based
# interpretation because this was common practice at the time and
# existing implementations had made the fileid or equivalent available
# for judicious application use.  Some uses are consistent with the
# intent of the Standard.  Others are not, and as always the standard
# has made no effort to enumerate every possible mode of abuse.
#
# For example, it is obvious that while the fileid given INCLUDE-FILE
# is certainly known to the application, it would be imprudent for the
# application to use CLOSE-FILE on that fileid during its use by the
# interpreter, and the fact that the Standard does not explicitly
# forbid this action does not in any way diminish its absurdity.
#
# In this light the individual points are addressed as follows:
#
# > What must happen if file operations are used with the value returned
# > by SOURCE-ID ?
#
# Passive file operations such as FILE-POSITION and FILE-SIZE are
# appropriate.
#
# Reading and repositioning operations are appropriate and were
# specifically envisioned by those who drafted the file word sets,
# although these activities are of course environmentally dependent
# upon having a file as the input source and in many cases (such as
# sequential files like tapes, or temporally disjoint files such as
# terminal devices, named pipes, or network connections) the very
# ability to reposition is environmentally dependent on the *type*
# of file.  It is recognized that the ability to perform these oper-
# ations constrains implementations, but such practices were common
# and hence no effort was made to prohibit them.
#
# The "synchronization" of the text interpreter with other explicit
# file operations opens the door to useful techniques such as files
# that begin with Forth code whose purpose is to operate on data that
# follows the code in the file.
#
# Any type of modification operations on the file itself,  such as
# writing, resizing, or deletion, and on the fileid, such as closing,
# is either prima facie absurd (such as closing) or depends on "carnal
# knowledge" of the implementation and is hence environmentally depen-
# dent on things the Standard doesn't require that implementations
# document.  Furthermore the ability to write into a "file" is in
# itself environmentally dependent on the characteristics of the file
# and the way in which it was opened.  Such operations may be useful
# but are not portable and are beyond the scope of the Standard.
#
# As a decision making example, it is without question a *useful*
# technique for one part of a program to inject source code into a
# named pipe while another part is interpreting from the other end
# of that pipe.  However a program that depends on operating system
# support for named pipes is certainly not standard!
#
# > How INCLUDE-FILE and INCLUDED interfere with other file access words?
# >
# > In other words:
# > >11.6.1.1718 INCLUDED                                 FILE
# > >     ( i*x c-addr u -- j*x )
# > > ...
# > >Repeat until end of file: read a line from the file, fill the input
# > >buffer from the contents of that line, set >IN to zero, and interpret.
# > > ...
# >
# > Does the word "read" imply the functionality of READ-LINE ?
#
# Yes, that is exactly what it means.
#
# (11.6.1.2090, the definition of READ-LINE, is the only definition
# the standard offers for "reading a line from a file" and is by
# definition what the phrase cited above refers to.  "line" is
# defined in 2.1 in a way that is consistent with 11.6.1.2090.
#
# > At the conclusion of the operation ("read"), what the value returned
# > by FILE-POSITION shall be?
#
# Refer to 11.6.1.2090 READ-LINE which defines this value.
#
# > Are implementors allowed to let their systems read the whole file
# > into a buffer?
#
# Yes.  Many systems cache files in memory.
#
# > The ambiguity springs from the fact that INCLUDE-FILE may read lines
# > one by one or load the whole file into a buffer and take the lines
# > from there.
#
# There is no ambiguity.  An implementor may cache a file if desired
# but must implement file caching in such a way that it is still
# accessed through the normal file operators.  This is common practice
# throughout the computer industry.  Caching or any other optimization
# that is done in a transparent way is entirely consistent with the
# Standard.
#
# An ambiguity only arises if an implementor forgets that transparency
# is required of him, which would be an error since transparency is
# a paramount concern when conforming to standards or seeking portability.

   ------ the question ----------------------------------------------------

   What must happen if file operations are used with the value returned
   by SOURCE-ID ?
   How INCLUDE-FILE and INCLUDED interfere with other file access words?

   In other words:
   >11.6.1.1718 INCLUDED                           FILE
   >       ( i*x c-addr u -- j*x )
   > ...
   >Repeat until end of file: read a line from the file, fill the input
   >buffer from the contents of that line, set >IN to zero, and interpret.
   > ...

   Does the word "read" imply the functionality of READ-LINE ?
   At the conclusion of the operation ("read"), what the value returned
   by FILE-POSITION shall be?
   Are implementors allowed to let their systems read the whole file
   into a buffer?

   ------------------------------------------------------------------------

   The ambiguity springs from the fact that INCLUDE-FILE may read lines
   one by one or load the whole file into a buffer and take the lines
   from there.

   I see several (currently, 5) possible variants of the answer, the best
   one IMHO is the 4th. Note that the variants 1-4 allow the use of buffers
   for the files being INCLUDEd, while the 5th does not.

   ---- VARIANT 1
   ----------------------------------------------------------------
   > 11.6.1.2218 SOURCE-ID                         source-i-d      FILE
   >       ( -- 0 | -1 | fileid )
   > ....
   Ambiguous conditions:
   - fileid is used with other file access operations
   - a program changes the file specified by fileid via a different file
     identifier

   Rationale. The system implementors are free to read the source file
   line-by-line or to load it into a buffer (the whole file or a part of
   it). The position at which the file pointer will be, as well as the
   effects of changing it, are unpredictable in the general case.

   ---- VARIANT 2
   ----------------------------------------------------------------
   > 11.6.1.2218 SOURCE-ID                         source-i-d      FILE
   >       ( -- 0 | -1 | fileid )
   > ....
   Ambiguous conditions:
   - a program performs a file access operation on the file specified by
     fileid using the current position of the file
   - a program changes the current position of the file specified by
     fileid and does not restore it before the system attempts to refill
     the input buffer
   - a program changes the file specified by fileid

   Rationale. The system implementors are free to read the source file
   line-by-line or to load it into a buffer (the whole file or a part of
   it). The position at which the file pointer will be, as well as the
   effects of changing it, are unpredictable in the general case.

   Note: the input buffer may be refilled at the end of line or due to
   execution of RESTORE-INPUT or other word temporary changing the input
   source.

   ---- VARIANT 3
   ----------------------------------------------------------------
   > 11.6.1.2218 SOURCE-ID                         source-i-d      FILE
   >       ( -- 0 | -1 | fileid )
   > ....
   Ambiguous conditions:
   - a program performs a file access operation on the file specified by
     fileid using the current position of the file
   - a program changes the current position of the file specified by
     fileid and does not restore it before the system attempts to refill
     the input buffer.
   - a program changes the already existing portion of the file
     specified by fileid
   - a program changes the file specified by fileid via a different file
     identifier

   Rationale. The system implementors are free to read the source file
   line-by-line or to load it into a buffer (the whole file or a part of
   it). The position at which the file pointer will be, as well as the
   effects of changing it, are unpredictable in the general case.
   On the other hand, if new text is added to the end of file, the
   system stil can determine it and interpret the text.

   Note: the input buffer may be refilled at the end of line or due to
   execution of RESTORE-INPUT or other word temporary changing the input
   source.

   ---- VARIANT 4
   ----------------------------------------------------------------
   > 11.6.1.2218 SOURCE-ID                         source-i-d      FILE
   >       ( -- 0 | -1 | fileid )
   > ....
   The current position for the file identified by fileid will correspond
   to the beginning of the next source line (the line immediately following
   the system-defined enf-of-line indicator of the line being currently
   interpreted with the text interpreter).

   Ambiguous condition:
   - a program changes the line being currently interpreted in the file
     specified by fileid.
   - a program changes the file specified by fileid via a different file
     identifier

   Rationale. The system implementors are free to read the source file
   line-by-line or to load it into a buffer (the whole file or a part of
   it). In the latter case, the system must check if the fileid passed
   to the file access operations corresponds to one of the files being
   currently loaded (INCLUDEd), and behave as if the file was read
   line-by-line. If the program changes the file specified by fileid,
   the contents of the buffer must be adequately changed. Implementing
   a buffer for file loading is equivalent to implementing a cache.
   Reading/writing to the cached file bypassing this cache (e.g. by
   means of operating system of firmware) also leads to
   unpredictable results, but the only "standard" way of bypassing the
   cache is using two or more different file ideentifiers for the same
   file. The statement that changing the line being currently interpreted
   is ambiguous means that the systems (especially those that read files
   line-by-line) are allowed not to reload the current line when the file
   changes, but this may happen as a result of execution of SAVE-INPUT
   and RESTORE-INPUT.

   ---- VARIANT 5
   ----------------------------------------------------------------
   All what is written above is wrong (below ":" marks citations
   from  'variant 4'). We see in 11.6.1.1718 INCLUDED :

   >11.6.1.1718 INCLUDED                           FILE
   >       ( i*x c-addr u -- j*x )
   > ...
   >Repeat until end of file: read a line from the file, fill the input
   >buffer from the contents of that line, set >IN to zero, and interpret.
   > ...

   The words "read a line" above mean "read as with READ-LINE", not
   "get somehow the next line into the input buffer, for example,
   pre-read the whole file into a buffer and then take a line from
   that buffer".
   This means that a standard system is not allowed to read the whole
   file into a buffer and then take lines from there. As a result,

   :The current position for the file identified by fileid will correspond
   :to the beginning of the next line (the line immediately following
   :the system-defined enf-of-line indicator of the line being currently
   :interpreted with the text interpreter).

   and two comditions
   :Ambiguous condition:
   :- a program changes the line being currently interpreted in the file
   :  specified by fileid.
   :- a program changes the file specified by fileid via a different file
   :  identifier
   should be, indeed, ambiguous. The first one is ambiguous because
   a standard system may or may not reload the current line when the file
   is changed and the words SAVE-INPUT and RESTORE-INPUT execute.
   The second one should be ambiguous because the result would depend
   on the operating system and loaded drivers.

   -------------------------------------------------------------------------------

   Note that variants 4 and 5 are equivalent.
